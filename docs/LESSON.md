# Lessons

## 2026-07-13 (Macro D, D-PR5 webhook trigger)

- CORRECTED (round-1 review of THIS entry — a reviewer caught the original claim as an unverified overclaim, matching this session's own "verify framework behavior from source, not memory" rule): Laravel 13's `route:cache` does NOT reject a closure route action outright — `Illuminate\Routing\Route::prepareForSerialization()` (called by `RouteCacheCommand::handle()` for every route) runs `serialize(SerializableClosure::unsigned($this->action['uses']))`, which CAN successfully serialize a plain closure (confirmed by reading both files directly; there is no `LogicException`/"Uses Closure" throw in this command for Laravel 13, contrary to an initial competing claim during review — that behavior may be true of an older Laravel version, not this one). The REAL problem was narrower and worse: the route action was an ARROW FUNCTION implicitly bound to `$this` (the registrar instance), so serializing it drags the registrar's WHOLE bound object graph (container, cache repository, trigger service) into the cache too — which can either fail serialization outright (if anything in that graph is genuinely unserializable) or, more insidiously, SUCCEED and bake STALE service instances into a production route-cache file that persists across the app's lifetime until the next `route:cache` run — no error surfaces either way, so this class of bug is easy to ship undetected. The fix — an invokable controller CLASS as the action with per-route data threaded through `Route::defaults($key, $value)` instead of closure captures — avoids the bound-object-graph problem entirely regardless of whether serialization would have "worked": a container-resolved controller re-fetches its dependencies fresh on every request, never freezing them into a cache file. Lesson generalizes: before writing route-cache/config-cache guidance into LESSON.md (or any durable doc), grep the ACTUALLY INSTALLED framework version's source for the specific method/exception named, the same discipline already applied to runtime-behavior claims elsewhere in this file — a plausible-sounding framework claim from memory (mine OR a reviewer's) can be wrong in either direction.
- A `Route::defaults()` value is written verbatim into the SAME route-cache build artifact as the URI/action — so a SECRET must never be threaded through it, even though defaults are the correct mechanism for route-cache-safe non-sensitive data (like a slug). The safe pattern: thread only a look-up KEY (slug) through the default, and have the controller re-resolve the sensitive value fresh from config/a repository at request time. This is a stronger constraint than "must be route-cache-serializable" — a value can be serializable and still be the wrong thing to persist into a build artifact.
- `Route::parameter($name)` merges in defaults for keys that were NEVER part of the URI's `{placeholder}` pattern at all (confirmed by reading `Illuminate\Routing\RouteParameterBinder::replaceDefaults()`'s second loop, which copies EVERY `$route->defaults` key into `$parameters` if not already set) — so `Route::defaults()` works as a general-purpose "attach arbitrary data to this route, retrievable via `parameter()`" mechanism, not only as a way to supply a default for an OPTIONAL URI segment. Useful for passing config alongside a route without polluting the URI pattern itself.
- Testing a request handler by calling `Router::dispatch($request)` directly (bypassing the real HTTP `Kernel::handle()`) breaks a route closure/controller's own `Request $request` type-hinted parameter resolution: Laravel resolves that parameter from the CONTAINER's `request` binding (`app('request')`), not from the object passed into `dispatch()`. The real kernel rebinds `app()->instance('request', $request)` before routing; a test driving `Router::dispatch()` directly must do the same rebind manually, or the handler silently sees Testbench's original bootstrap request (missing whatever headers/body the test just built) instead of the one under test — this produced a confusing "signature missing" failure that had nothing to do with the signature-verification code itself.
- The "never let an internal failure escape to the caller" isolation lesson from D-PR4 (event listeners) generalizes with a further subtlety for D-PR5 (webhook controller): a strictly-typed helper parameter (e.g. `stringParameter(Route $route, ...)`) can ITSELF `TypeError` on a legitimately-nullable upstream value (`Request::route()` can return `null`) BEFORE the caller's own try/catch runs — the exact same bug SHAPE as D-PR4's typed-closure-parameter finding, but at a completely different call site (a private helper method, not an event listener). This suggests a general audit heuristic: for any class whose whole justification is "wrap everything, nothing escapes," grep every method signature for non-nullable typed parameters fed by a framework value the docs/source say CAN be null, not just the obvious outermost entry point.
- When a class's docblock makes a strong claim ("wraps its ENTIRE body in try/catch") and a reviewer finds the claim doesn't quite match the code, the reflex should be "which is actually correct, the code or the docblock?" — not automatically weaken the doc to match looser code. Here moving the pre-try-block statements INSIDE the try was strictly safer (closes a real defensive gap) than just rewording the comment, and cost nothing extra since the values were already being computed unconditionally either way.

## 2026-07-13 (Macro D, D-PR4 event trigger)

- A listener registered via `Dispatcher::listen($eventClass, $closure)` must NOT strictly type its parameter as the event object (`function (object $event)`): Laravel's dispatcher invokes a listener with `array_values($payload)` when an event fires via the legacy string-name-plus-payload form (`Event::dispatch($eventClass, $payload)`), not necessarily a single typed object. A type mismatch raises a `TypeError`/`ArgumentCountError` BEFORE the closure body — and therefore before any `try/catch` inside it — runs, so a package promising "this trigger's failures never escape into the host's call stack" must make the closure's own parameter list untyped/variadic and validate the actual payload shape INSIDE the try, converting every misshapen dispatch into a caught, logged skip instead of an escaping fatal. Caught by Codex review, not by initial testing (the happy-path test only ever dispatched via the modern `new EventClass(...)` object form).
- A boot-time "does this config value implement interface X" check using `is_a($class, Interface::class, true)` is NOT sufficient when the value will later be CONTAINER-INSTANTIATED: `is_a()` accepts an interface name or an abstract class name too (anything that "is a" the target type structurally), both of which pass this check but can never actually be built by `$container->make()`. The gap degrades a single config mistake from "one skip logged once at boot" into "the same warning logged on every matching event/trigger firing forever." Add `(new ReflectionClass($class))->isInstantiable()` alongside the `is_a()` check for any config value destined for container resolution, not just for a type-implements check.
- The core isolation principle from this PR generalizes to D-PR5 (`WebhookTrigger`): a SCHEDULED task runs in a process this package effectively owns; an EVENT listener runs inside the HOST APPLICATION's own event-dispatch call stack; an INBOUND WEBHOOK handler runs inside the host's own HTTP request-response cycle. Each has an analogous but distinct "must never let an internal failure escape and break something else" requirement — for the webhook trigger specifically, that means returning a proper HTTP error response (401 on bad signature/timestamp, etc.), never letting an internal exception surface as an unhandled 500 or crash the request.

## 2026-07-13 (Macro D, D-PR3 schedule trigger)

- A reviewer's claim about FRAMEWORK behavior can be stale/wrong even when it describes something that was once true. Round-6 review on `laravel-flow-connect` PR #3 asserted `php artisan schedule:cache` excludes closure-based `Schedule::call()` events from the cache — real behavior in OLDER Laravel versions, but `grep -rln "schedule:cache\|ScheduleCache" vendor/laravel/framework/` against the ACTUALLY installed Laravel 13 returned nothing: the command doesn't exist in this version (only a vestigial `ScheduleClearCacheCommand`, unrelated — clears mutex state, not a schedule-definition cache). Declined with the grep proof rather than adding a README caveat for a limitation that cannot be reproduced. Same discipline as the `FlowEngine::dispatch()` docblock lesson below: verify a specific-sounding framework claim against the installed `vendor/` source for the ACTUAL target version, not against general Laravel knowledge that may predate a breaking change.
- A package's own service provider resolving a framework container singleton EAGERLY in `boot()` can force unrelated work for every process that boots that provider. Resolving `Illuminate\Console\Scheduling\Schedule::class` directly (even behind a `runningInConsole()` guard) still fires on EVERY console command the package ships alongside — `migrate`, `queue:work`, `tinker` — not just the scheduler commands, because `runningInConsole()` is true for all of them and resolving that singleton immediately triggers `ConsoleKernel::resolveConsoleSchedule()`, which builds the HOST APPLICATION's entire schedule (every cron entry the app defines, not just the package's own). Fix: `$this->app->afterResolving(Schedule::class, function (Schedule $schedule) { ... })` registers a callback that only fires if/when something else in the process actually resolves that singleton — the standard Laravel pattern for a package contributing schedule entries without forcing schedule construction itself. Pin the behavior with a regression test asserting `$this->app->resolved(Schedule::class)` is `false` right after boot.
- `??` (null-coalescing) collapses BOTH a missing array key AND an explicitly-`null` value to the same default — `$entry['input'] ?? []` treats `['input' => null]` identically to omitting the `input` key entirely. A reviewer flagged this as bypassing later `is_array()` validation, but that's the CORRECT behavior when `null` and "omitted" are meant to be synonymous ("no input provided", a common config idiom) — not a bug to fix by rejecting explicit `null`. When declining a "coalesce hides malformed input" finding, check whether the collapsed default is itself a VALID value for that field (here, `[]` genuinely is valid `input`) — if so, no value can actually reach downstream code in a malformed state, and the finding conflates "collapses two representations of the same intent" with "silently accepts bad data."

## 2026-07-13 (Macro D, D-PR2 architecture correction)

- Documenting a Laravel framework behavior (queue dispatch return values, `ShouldQueueAfterCommit` semantics, transaction-deferral guarantees) from PLAUSIBLE-SOUNDING MEMORY rather than verified source took FOUR review rounds to converge on `src/Contracts/FlowTrigger.php`'s one paragraph about what `FlowEngine::dispatch()` returns: "sync→FlowRun, async→null" (wrong, unverified), "always after-commit-deferred regardless of driver" (overstated — `ShouldQueueAfterCommit` only defers when a transaction is actually open), "queues a run" (wrong — the `sync` driver runs inline, no queue involved). The pattern that finally worked: state ONLY what's directly verifiable by reading the actual source in this repo (`RunFlowJob implements ShouldQueueAfterCommit` — a one-line `grep`-able fact) plus the return TYPE (`mixed`, from the method signature), and explicitly decline to assert anything about runtime VALUE/behavior that would require certainty about Laravel's internal queue dispatcher across driver/transaction combinations. For any `@api` docblock describing a framework interaction's runtime behavior: either verify the specific claim against source/tests, or scope the claim down to what's true unconditionally — don't pad an accurate core fact with a plausible-sounding elaboration.

- When a per-macro plan file (authored by an earlier session, e.g. `docs/superpowers/plans/2026-07-12-macro-d-realtime-triggers.md`) paraphrases an architectural decision from the AUTHORITATIVE master plan, VERIFY the paraphrase against the master plan's own exact wording before implementing — don't trust a derived doc's summary as ground truth for a cross-repo placement decision. The per-macro plan's D-PR2 section said only "a `Contracts/` namespace" without specifying WHICH repo; the master plan (`docs/superpowers/plans/2026-07-07-flow-v2-program-master-plan.md`) is unambiguous ("trigger contracts in core; ... trigger contracts consumed from core"). Implementing against the derived plan's vaguer wording put the `FlowTrigger` interface in the satellite `laravel-flow-connect` package instead of core — a real architectural mistake requiring 3 follow-up PRs across 2 repos to fix (core PR #73 to add it correctly, connect PR #2 to remove the duplicate and consume core's version, plus this docs correction) after a reviewer (Codex) caught it by cross-referencing the master plan directly. When a per-macro plan is ambiguous about WHERE something goes in a multi-repo program, re-read the master plan's own sentence, don't infer.
- A cross-repo dependency fix has a THIRD location that's easy to forget: not just the two repos' code, but the DEV-TIME repository/branch pin connecting them. Fixing core (add contract) and connect (consume it) still left connect's `composer.json`/CI pinned to `main`/`dev-main`, which does NOT have the fix yet (it lives on core's `task/v2d-realtime-triggers` macro branch until Macro D's gate merges it) — had to retarget connect's dev dependency to `dev-task/v2d-realtime-triggers` temporarily, mirroring the same "subtask branches point at the macro branch, not `main`" rule already used WITHIN core, but now applied ACROSS repos. Document the retargeting stages explicitly (macro branch → main → tagged release) so a future session doesn't ship connect pointed at a core ref that's missing in-progress cross-repo work.

## 2026-07-13 (Macro D, D-PR2 laravel-flow-connect bootstrap)

- Composer's `repositories` block is read ONLY from the ROOT package of an install — a package's own `composer.json` declaring a path/VCS repository for one of its dependencies is honored while developing ON that package, but is SILENTLY IGNORED when the package is installed AS a dependency by a host app. A package that requires an unreleased sibling (as `laravel-flow-connect` does for `padosoft/laravel-flow` pre-tag) is therefore genuinely not `composer require`-able by anyone outside the monorepo-adjacent dev setup, until the sibling tags. Document this explicitly in the README (with the exact repository block a host app needs) rather than hiding it — a reviewer (Codex) caught this being silently glossed over.
- Composer stability flags do NOT propagate through the dependency graph: an explicit `dev-main`-style suffix on a DIRECT root requirement only affects stability resolution for THAT package, not its own transitive dependencies. Requiring `pkg-a:dev-main` at the root does not, by itself, permit `pkg-a`'s own `pkg-b:dev-main` requirement to resolve if the root's `minimum-stability` is `stable` — the root needs `"minimum-stability": "dev"` (+ `"prefer-stable": true` to still prefer stable versions where available) to cover the whole subtree, or must explicitly re-list every dev-stability transitive package itself. Got this wrong twice in a row in README prose before verifying the propagation rule carefully (Copilot caught both times) — when documenting Composer stability behavior for a REAL scenario, state the propagation rule explicitly rather than asserting a shortcut "just add `:dev-main` to your require line" that only covers the direct-package case.
- A PHPStan `trait.unused` finding (`level 8`, "Trait X is used zero times and is not analysed") is a legitimate signal to REMOVE the trait, not suppress the check — if a shared-implementation trait has no concrete consumer in `src/` yet (only anonymous classes in tests), it's a premature abstraction per this program's "three similar lines is better than a premature abstraction" rule. Defined a `FlowTrigger` contract interface with NO default-implementation trait in D-PR2 for exactly this reason — a future PR with 2-3 real concrete triggers sharing the identical one-line `fire()` body is the right time to extract a trait, not before the first consumer exists.

## 2026-07-13 (Macro D, D-PR1 broadcasting opt-in)

- Calling `Event::fake()` a SECOND time mid-test silently breaks any already-resolved container SINGLETON that constructor-injected the `Dispatcher` (e.g. this PR's `GraphProgressBroadcaster`, resolved once and cached on first use). `Event::fake()` rebinds the container's `Dispatcher`/`'events'` to a brand-new `EventFake` instance each call — it does not mutate/clear the existing one — so a singleton built during an EARLIER part of the test keeps its OLD (now orphaned) fake reference and its dispatches never reach the second fake's `Event::dispatched()`/`assertDispatched()` calls. Symptom: dispatches from the FIRST phase of the test are captured fine, dispatches from a SECOND phase (after a second `Event::fake()`) come back silently empty — no exception, easy to misdiagnose as a real bug in the code under test. Fix: call `Event::fake()` ONCE per test and distinguish separate scenarios some other way (e.g. by a run id embedded in the payload) rather than re-faking mid-test. This is a TEST-ONLY gotcha — production `Illuminate\Events\Dispatcher` is never swapped mid-request, so it cannot bite outside a test.
- No single physical call site persists RUN-level state for both graph engines (unlike node-level state, which funnels through one shared `NodeExecutor::persist()`): `GraphRunner::persistRunFinished()` (sync) and `QueueGraphCoordinator::finalizeRun()` (queued) are two independent methods that each independently call `RunRollup::state()`/`RunRollup::counters()` then `$store->runs()->update(...)`. Rather than force them into one physical seam (which the current architecture doesn't support without a larger refactor), broadcasting follows the C-PR8 `GraphSaga` precedent: ONE shared service (`GraphProgressBroadcaster`) both runners call into with the ALREADY-shared `RunRollup` computation, so the "enabled" gate and payload-building logic live in exactly one place even though there are two call sites. "One seam" should mean "one place the LOGIC lives," not necessarily "one physical call site," when the codebase's actual architecture has two independent finalization paths.
- Broadcasting was deliberately DECOUPLED from `laravel-flow.persistence.enabled`: `NodeExecutor::persist()`'s node-transition broadcast fires even when `$store` is null (persistence disabled), gated only on `! $dryRun`. A dry run always stays silent (a simulation broadcasting fake progress would violate its "zero externally-observable side effects" contract), but persistence-off + broadcasting-on is a legitimate combination — the run id/node id/state are known regardless of whether anything gets written to `flow_run_nodes`.
- Round-1 review (Codex) caught two real gaps the first cut of `persist()`/`finalizeRun()` missed:
  1. **A broadcast dispatch must NEVER be allowed to propagate an exception into the executor.** The initial version called `$this->events->dispatch(...)` uncaught inside `NodeExecutor::persist()`, BEFORE the durable DB write. A throwing broadcast driver would then abort `persist()` before `flow_run_nodes` got updated — on the queued path the job retries, re-executing a handler that ALREADY succeeded (silent double-execution of side-effecting work). Fix: `GraphProgressBroadcaster` now wraps every dispatch in try/catch (`Log::warning`, class+code only — same discipline as the node-cache/approval-token best-effort writes elsewhere in `NodeExecutor`), AND `persist()` was reordered to write the DB row FIRST, broadcast SECOND — defense in depth on top of the internal catch, not a substitute for it.
  2. **Broadcasting must never run SYNCHRONOUSLY while holding the `flow_runs` row lock.** The initial `QueueGraphCoordinator::finalizeRun()` called the run-progress broadcast from INSIDE `advance()`'s `lockForUpdate()` transaction. A slow/failing broadcast driver would then stall every duplicate coordinator/join waiting on that lock for a reason that has nothing to do with the lock's actual purpose (serializing node claims). Fix: `finalizeRun()` no longer broadcasts; `advance()` re-reads the just-committed run row AFTER the transaction closure and broadcasts from that (reusing the DB's own committed truth instead of re-deriving state outside the lock that produced it) — same "outside the lock, after commit" pattern already established for `GraphSaga` compensation and the parent-join drive in this file.
  3. Related: blocked nodes (`GraphRunner::persistBlocked()` / the queued coordinator's poison-propagation loop) never went through `NodeExecutor::persist()` — the ONE seam every other node-transition broadcast piggybacks on — so they broadcast nothing while the run-level aggregate snapshot silently counted them as failures. Both call sites now collect blocked transitions and broadcast them explicitly (in the queued path, collected under the lock but broadcast AFTER it — same pattern as above).

## 2026-07-13 (Macro C acceptance gate, G3.2)

- Found while writing the combined "nested, fan-out, approval-gated graph on queue" acceptance test: `JoinCoordinator::childCompleted()` calls `$this->childRunner->spawnNext($parentRunId, $parentNodeId)` WHILE STILL HOLDING its own join cache lock (`laravel-flow:join:{runId}:{nodeId}`, released only in the surrounding `finally`). Under a real async queue driver this is harmless — `spawnNext()`'s dispatch just enqueues a row, nothing executes inline. But under the FULLY SYNCHRONOUS/RECURSIVE `sync` queue driver (used extensively elsewhere in this test suite — `CoordinatorRaceTest`, `CoordinatorRecoveryTest`, `GraphApprovalResumeTest` all rely on it as a first-class, legitimate testing/deployment mode) with `maxConcurrency > 1` (a burst spawns more than one child in the SAME call), a newly-spawned child can run-and-complete SYNCHRONOUSLY before `spawnNext()` even returns — and that child's OWN completion recursively re-enters `childCompleted()` for ITSELF, trying to acquire the SAME (still-held, single PHP thread) lock key. A single-threaded process can never release a lock it is still holding further up its own call stack while blocked waiting for it, so `$lock->block()` deterministically times out (`LockTimeoutException`), failing the fan-out node. Confirmed with a minimal `flow.foreach` (`maxConcurrency: 2, items: [1,2]`) run via `dispatchGraph()` on `queue.default = sync` — the FIRST (non-duplicate) execution already fails this way, before any deliberate duplicate-dispatch simulation. `ChildFlowRunner::claimNextPending()`'s OWN DB `lockForUpdate` claim is the real exclusion primitive for child-slot spawning (per its docblock), so `spawnNext()` does not actually NEED the join cache lock held around it — the fix is to call it AFTER releasing the join lock, but doing that safely requires re-deriving the subsequent `countUnfinished()`/`resumeParent()` decision without either double-resuming the parent (a recursive spawn could resolve everything before the outer frame checks again) or reacquiring the SAME lock (which reintroduces the identical risk if the recursive cascade is still in flight) — a genuine redesign of this lock's critical section, not a safe same-round patch alongside everything else already changed in `JoinCoordinator` this gate. Worked around for the acceptance test itself by using the REAL database queue driver + `queue:work --once` draining (the proven `ControlNodeQueuedTest` pattern — each `Artisan::call()` invocation still runs in the SAME PHP test process, but pulls its job from the DB queue table rather than executing it inline/recursively, so the self-deadlock's trigger never occurs) instead of `sync`, which is what any genuine multi-worker deployment would use for fan-out anyway. Tracked as a follow-up: harden `JoinCoordinator::childCompleted()` against sync-driver recursive self-deadlock before recommending `QUEUE_CONNECTION=sync` for any deployment using `maxConcurrency > 1` fan-out/sub-flow nodes.

## 2026-07-12 (Macro C gate, PR #66)

- Non-blocking but worth a future hardening pass: `NodeExecutor`'s retry-backoff loop calls `Sleep::for($backoff)->seconds()` on EVERY path, including when `$queued === true` — so a queued node's retry blocks the queue WORKER process for the whole backoff window instead of releasing the job back to the queue (Laravel's usual `release($delay)` pattern, already used elsewhere in this codebase — see `JoinCoordinator`'s LESSON entry above and `NodeJob`). For a large `#[Retry(backoff: ...)]` this can noticeably reduce worker throughput, and if the backoff exceeds the worker's own `--timeout`, the job could be force-killed mid-sleep, leaving the node's claim in an inconsistent state. It is NOT a regression introduced by Macro C's later PRs (C-PR4 shipped it this way) and it does not break correctness today (retries genuinely happen, dead-letter still works) — it is a throughput/robustness trade-off. The proper fix restructures the retry loop into a RESUMABLE state machine on the queued path: persist the failure + `available_at` (already done), have `NodeJob` `release($backoff)` instead of looping in-process, and on redelivery reconstruct the attempt count from the persisted `flow_run_nodes.attempts` column rather than an in-memory loop variable — a genuine redesign of the "one seam, two callers" retry loop shared with the synchronous path, not a safe same-round patch. Deferred as a tracked follow-up (see PR #66 review thread) rather than rushed.

- A graph run started SYNCHRONOUSLY (`GraphRunner::run()`) only ever persists a `flow_run_nodes` row for a node once it actually becomes ready — unlike a run started via `QueueGraphCoordinator::start()`, which pre-seeds EVERY node `pending` up front (so its CAS-based `claim()` always has a row to flip). This is invisible until something hands a sync-started run off to the QUEUED advancement machinery for the first time (C-PR10's approval-gate resume does exactly this: mark the paused node's row and dispatch `CoordinatorJob`) — `claim()`'s `WHERE status='pending'` silently affects ZERO rows for any node that was never reached, so downstream nodes are computed as `ready` by `ReadinessResolver` but never actually get claimed/dispatched, and the run finalizes early as `partially_succeeded` with those nodes stuck `Pending` forever. Symptom is subtle: no exception, no error — just a `Pending`-forever node and a wrong terminal run state. Fix: whoever performs that hand-off must pre-seed any missing `pending` rows first (mirroring `QueueGraphCoordinator::start()`'s seeding loop) before dispatching into the queued path. Caught only by writing an actual synchronous-run-then-resume integration test and observing the downstream node's invocation count stay 0 — a unit test of the coordinator alone (which always starts pre-seeded) would never surface this.
- SQLite's (and most DBs') `INSERT ... ON CONFLICT DO UPDATE` still validates NOT NULL columns on the proposed INSERT row BEFORE evaluating the conflict target — so a "partial-key upsert" (`RunNodeRepository::createOrUpdate()`, whose `DO UPDATE` clause only touches the keys you pass) still needs EVERY NOT NULL column supplied on every call, even one that is logically "just an update" of an already-existing row. On `flow_run_nodes` that's `node_type` alone (`sequence` is nullable in the migration; only `node_type` has no `->nullable()`) — but the general rule is "check the migration for which columns actually lack a DB default/nullable modifier," not "assume every column a call site conventionally passes is required." Omitting a genuinely-required one fails with a NOT NULL constraint violation on the INSERT half of the statement, not a silent partial write. Contrast with the C-PR7 lesson ("the upsert only writes provided keys") — that's about which columns get UPDATED on a conflict, not about what the INSERT attempt itself requires.
- The PHPStan version-skew lesson generalizes to the WHOLE vendor dir: laravel/framework 13.19 added an optional `$timeout` param to the `Illuminate\Contracts\Concurrency\Driver::run()` contract, so an anonymous-class test double written against the older signature FATALS ("Declaration must be compatible") only in CI — and paratest swallows the real message, reporting just "Premature end of PHP process". Diagnosis recipe: run `composer update` (full, not just phpstan — composer.lock is gitignored here) and re-run the failing test locally; the real fatal appears immediately. Prevention: when stubbing a FRAMEWORK contract, declare any extra optional params the newest framework added (an implementation may always add optional params over the interface, so the stub stays compatible with older versions too) — or extend the framework's own concrete driver instead of re-implementing the interface.
- CI runs `composer update` fresh while the local vendor dir ages, so CI's PHPStan can be a NEWER release with stricter inference — a locally-green `composer analyse` can fail in CI on rules the local version doesn't have yet (bit C-PR8: PHPStan 2.2.x narrowed a `@var array<int, array{success: bool,...}>` result union to *NEVER* where 2.1.x accepted it). Two habits: (1) when CI PHPStan fails on code that passed locally, first compare `vendor/bin/phpstan --version` against the CI log and `composer update phpstan/phpstan larastan/larastan --with-dependencies` locally (composer.lock is gitignored here, so this is safe) to reproduce; (2) prefer honest wide types (`array<int, mixed>` + runtime guards) over optimistic array-shape docblocks for data crossing an external boundary (a Concurrency driver, a queue payload) — shapes you don't actually control invite exactly this class of version-sensitive false narrowing.
- `NodeResult::paused()` carries `success === true` (it mirrors a v1 paused step: the node succeeded at *reaching* the pause, it just awaits external input). Any executor branch keyed on `$result->success` therefore ALSO fires for paused — including the node-cache WRITE. Caching a paused node's partial output then serves a later run as a completed `succeeded`, silently skipping the pause. Guard cache writes (and anything that treats "success" as "final output ready") on `$result->success && ! $result->paused`. `dryRunSkipped` also sets `success === true`, but it can't reach the cache write because `$contentHash` only becomes non-null inside the `! $dryRun` read block — paused is the one live-run path that slips through. Copilot caught this; the fix is a one-token guard plus a `#[Cacheable]` fixture that pauses and asserts a second run re-runs the handler (0 cache rows).
- A `ContentHasher` built on `json_encode(..., JSON_THROW_ON_ERROR)` does NOT fail on objects: a plain object serializes to `{}` (or its public props) WITHOUT throwing, so two distinct objects collide to the same content hash and return a wrong cached result — reachable whenever a `PortType::Any` input/config carries an object. `JSON_THROW_ON_ERROR` only throws on resources/NAN/INF/malformed UTF-8, not on objects. Fail fast in `canonicalize()` (throw a dedicated `UnhashableInputException` on `is_object() || is_resource()`, reached at any nesting depth) so the caller's `try/catch` around `hash()` skips caching for that node and degrades to a normal uncached run — never a silent wrong hit.

## 2026-05-04 - Approval/Webhook Macro

- A paused approval step should not contribute to aggregate run output/business impact; it is control-plane state, not a completed business step.

- A paused approval step needs a matching Laravel event after `FlowStepStarted`; otherwise event subscribers observe a permanently running step.
- Approval gates should opt into dry-run execution by default so dry-runs stop at the same control-plane boundary as real executions.

- Approval approve/reject consumption should update by `token_hash` plus `pending` status in a single write, so repeated submissions cannot consume the same token twice.

- Negated `instanceof` guards should use explicit parentheses, e.g. `! ($value instanceof Type)`, so null-store checks remain clear to reviewers and future refactors.

## 2026-05-05 - Webhook Delivery Slice

- Webhook outbox repositories that previously required a `PayloadRedactor` constructor argument should keep an optional default binding via the service provider so direct resolvers still get JSON redaction; the engine layer can override per call with an execution-frozen redactor while the constructor default protects host apps that resolve the repository directly.
- Webhook outbox claim/lease pattern: pre-read candidates by `attempts < max_attempts` AND (status pending with available_at NULL/<=now OR status delivering with stale lease), then run a status-and-availability-conditional update that bumps attempts and pushes `available_at` into the future; concurrent claimers either lose the conditional update or see a future lease and skip.
- SQLite test fixtures that bypass Eloquent casts must pass `DateTimeInterface` instances (not ATOM-formatted strings) to query-builder `update()` because Laravel's `prepareBindings` formats `DateTimeInterface` to the grammar's date format (`Y-m-d H:i:s`); ATOM strings break later `WHERE column <= now()` comparisons because SQLite compares TEXT lexically and `T` > space.
- The webhook signature header follows Stripe-style convention `X-Laravel-Flow-Signature: t=<unix>,v1=<sha256>` where the HMAC signs `t.body` with the shared secret; emit the header only when a non-empty secret is configured so unsigned deliveries do not advertise a stale signature.
- Webhook outbox lifecycle covers `flow.completed`, `flow.failed`, `flow.paused`, and `flow.resumed`; compensated runs end as `flow.failed` for outbox purposes because the `flow.compensated` event is an internal transition rather than an external lifecycle signal.
- `flow:deliver-webhooks` retry uses configurable exponential backoff `(base * 2^(attempt-1))` and a configurable attempt cap; transient failures stay `pending` with a future `available_at`, terminal failures move to `failed` with `failed_at`, and successes move to `delivered` with `delivered_at` and cleared `available_at` / `last_error`.
- CLI delivery command options like `--batch` and `--sleep-ms` should validate as positive/non-negative integers via explicit numeric/regex guards rather than bare `(int)` casts so accidental string config (`--batch=abc`) returns a typed error instead of `Http::timeout(0)`-style silent zero.
- The default webhook URL config is the empty string, not `null`; aligning the README config block, config-table default cell, and actual `config/laravel-flow.php` keeps the docblock-vs-code contract consistent so reviewers and consumers do not infer a `null` default that the shipped code does not match.
- `markDeliveryResult` must be conditional on the row still being in `delivering` status so a competing process that already concluded the lease cannot be overwritten; the same conditional shape protects against duplicate command runs against the same record.
- `claimNextPending()` UPDATE should constrain on BOTH `status/available_at` AND `attempts = $candidate->attempts` so two concurrent processes that read the same row with the same attempts value cannot both win the compare-and-set; the status+available_at condition alone is insufficient if two workers observe stale-delivering rows that are still in delivering status.
- The outbox delivery lease should extend beyond the base backoff window by the configured `webhook.timeout_seconds` so an in-flight request cannot race with its own lease expiry; pass the delivery timeout to `nextLeaseAvailableAt` at claim time via `claimNextPending(now, deliveryTimeoutSeconds)`.
