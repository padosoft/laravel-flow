    1. `stream_set_timeout()`'s microseconds split used `round()` on the fractional remaining-time, which can hit exactly `1_000_000` — out of the function's valid `[0, 999999]` range with unpredictable cross-platform behavior.     2. The transport's raw response-line decode used a naive `is_array()` check after an associative `json_decode()`, which cannot distinguish a legitimate `{}`/object envelope from a malformed JSON ARRAY (`[]`/`[1,2]`) — both decode to the same PHP array shape. A malformed array response would silently pass as a valid-looking id-less notification and hang the caller until the overall deadline instead of failing fast. - **Task 13 (ledger + join)**: `flow_node_children` table (`@internal` `FlowNodeChildRecord`) is a per-item suspend/join ledger — one row per fanned-out child with a `pending -> running -> terminal` lifecycle, unique `(run_id, parent_node_id, child_index)` and unique `child_run_id` (nullable while pending), cascading off `flow_runs`. `Contracts\NodeChildRepository` (`@api`) + `Persistence\EloquentNodeChildRepository` (`@internal`): `recordPending` (idempotent), `claimNextPending` (atomic `lockForUpdate` claim of the lowest-index pending row — the sole mutual-exclusion primitive between the two spawn call sites), `attachChildRun`, `completeChild` (CAS `running -> terminal`), `countUnfinished`/`countRunning`, `forParent`. `Executor\JoinCoordinator` (`@internal`) resumes a suspended parent EXACTLY once when its last child terminates: under a per-parent cache lock it CAS-completes the child, releases the next pending item (windowing), and — only when zero children remain unfinished — flips the paused parent node to succeeded/failed with the ordered per-child `results` and returns a `JoinResult`. The join lock has its OWN small TTL (30s) and a separately-bounded block wait (5s), deliberately NOT the long node-execution `lock_seconds`.
- **Task 14 (control nodes, both engines)**: `@api` built-ins `flow.subflow` / `flow.foreach` / `flow.map` over a shared `@internal AbstractControlNode` + `ChildFlowRunner`. The node branches on the additive `NodeContext::$queued` (`@api`) execution-mode flag: **SYNC** (`GraphRunner`) runs each child inline via `runInline`, honoring `maxConcurrency` as a sequential BATCH size (documented, honest simplification — a synchronous runner cannot parallelize), aggregates ordered `results`, fails if any child failed; **QUEUED** (`QueueGraphCoordinator`) records every item as `pending`, spawns up to `maxConcurrency` as their own queued runs and suspends (`NodeResult::paused()`), and the join releases the next pending item per completion — TRUE capped concurrency, built rather than shipped as a documented gap (standing "most complete architecture" program rule). A ledger row is written on BOTH paths (a sub-flow is audit/Dashboard-visible either way), and a sync-vs-queued equivalence oracle test pins identical results. New nullable `flow_runs.graph` JSON column (additive migration `000011`) stores a queued run's canonical `GraphDefinition` UNREDACTED (flow STRUCTURE, not runtime payload — matching the `flow_definitions.graph` precedent; the redaction gate still covers `input`/`output`/`business_impact` only) so the queued join can reload a suspended parent and re-dispatch its `CoordinatorJob`.

- **Review**: remote GitHub Copilot + `chatgpt-codex-connector` — 4 rounds plus two independent manual re-reads, all findings fixed or declined-with-rationale. Real bugs caught and fixed: (1) an empty fan-out on the queued path paused forever (no child ⇒ nothing drives the join) — now returns `success(results:[])` immediately on both engines; (2) a spawn race — `spawnNext` originally read-then-dispatched-then-CAS'd (discarding the CAS result), so the control-node burst and the join release (under DIFFERENT lock keys) could dispatch the same pending item — fixed with claim-before-dispatch (`claimNextPending` inside the spawn transaction, mirroring C-PR5's `RunNodeRepository::claim()`); (3) a paused nested child prematurely joining its parent — gated `resumeParentIfChild` on the child being genuinely TERMINAL; (4) join lock TTL conflated with its block-wait (a contended worker could park for the whole TTL) — split into dedicated `lockSeconds`/`blockSeconds`; (5) non-idempotent `recordPending` threw on the unique constraint under retry, and a retried burst could over-spawn — made idempotent + added `countRunning`/`spawnNextIfUnderCap` so the burst tops up to the cap instead of double-bursting. - **Known non-blocking caveat** (see LESSON.md): `JoinCoordinator::childCompleted()`'s `block(5)` throws `LockTimeoutException` on contention with no explicit `tries`/backoff on `CoordinatorJob`, so retry-driven re-join reliability depends on the deployment's queue-worker `--tries` config — not a regression (the old `block(10)` had the same property; contention is brief by design) but a candidate for a future hardening pass toward `NodeJob`'s explicit `release($retry)` pattern.
