- **Design brief's information architecture** (read in full — `project/uploads/2026-07-07-flow-studio-ui-design-brief.md`, §5): `Sidebar: Overview / Flows (→ Studio, Versions&diff, Triggers, MCP exposure, Runs-of-this-flow) / Runs (→ Run detail = live monitor) / Approvals / Advisor (NEW) / Webhooks / Settings`, with ⌘K overlaying everything. §6 gives screen-by-screen requirements for all 11 screens (Overview, Flows library, Studio canvas editor, Dry-run/simulation, Run detail live monitor, Approvals inbox, Advisor, Triggers, MCP exposure, Runs/Webhooks/Settings lists, AI Flow Builder modal) — read this section again immediately before starting the E-PR that implements each screen, it is the authoritative prose spec the JSX/CSS visually encode.
- **Port-type color legend** (design brief §6.3, MUST match core's actual `PortType` enum — verify the exact 6 cases against `padosoft/laravel-flow`'s `src/Node/PortType.php` before wiring the frontend legend, do not trust the brief's suggested colors as gospel over the enum's real case list): text=sky, int/float=amber, bool=violet, json=emerald, any=neutral gray, binary=rose (brief's suggestion — confirm `PortType` has exactly these cases plus/minus any drift since the brief was written 2026-07-07, core has changed since then).
- **Node-state palette — VERIFIED, use these exact values, not the design brief's proposed names**: core's real `NodeState` enum (`src/Executor/State/NodeState.php`, `@api`, the single source of truth for the persisted `flow_run_nodes.status` string for both engines) has exactly 9 cases: `pending, running, paused, succeeded, failed, skipped, blocked, invalid_input, dead_letter`. The design brief (§6.5) proposes 10 states including `completed` (real value: `succeeded`), `paused-awaiting-approval` (real value: `paused` — the enum does not distinguish "paused for approval" from any other pause reason at the state-name level) and `cache-hit`, which **does not exist as a `NodeState` case at all** — a cache hit (core's `#[Cacheable]`/`NodeCache` mechanism) is metadata ON an otherwise-`Succeeded` node, not a distinct lifecycle state. E-PR5's live monitor must key its 9 primary state colors off the REAL enum values, and render "cache-hit" as a separate badge/overlay on a `Succeeded` node rather than as an 11th mutually-exclusive state — **verified gap**: the underlying Eloquent model (`Models\FlowRunNodeRecord`, `@internal`) HAS a `cache_hit` column, but core's `@api` `Dashboard\StepSummary` DTO (what `FlowDashboardReadModel::stepsForRun()` actually returns) does NOT expose it — `grep -n cache src/Dashboard/StepSummary.php` returns nothing. E-PR5 needs this field added to `StepSummary` first, which is CORE-repo work (its own branch/PR, per this grounding section's standing instruction below), not something `laravel-flow-admin` can work around by reaching past the `@api` DTO into the `@internal` model.
- **Graph/event JSON shapes given in the brief (§8)** are illustrative examples from the DESIGN brief, not verified against the CURRENT real API — before wiring E-PR2's canvas renderer or E-PR5's live monitor, re-derive the ACTUAL shapes from core's real contracts: `GraphSerializer::toArray()` (core, Macro B) for the graph envelope, and whatever broadcast event payload shape Macro D's `GraphBroadcastingTest` pins for live progress — do not trust the brief's example JSON as the literal wire contract.
- **`FlowDashboardReadModel` (core, `@api`)** is the SAME read contract this session's F-PR8 (`FlowAdvisor`) already builds on: `listRuns`, `findRun`, `pendingApprovals`, `listApprovals`, `listWebhookOutbox`, `failedWebhookOutbox`/`pendingWebhookOutbox`, `kpis()`. It does NOT expose a "list all flow DEFINITIONS" or "flow catalog" method (confirmed absent during F-PR5 grounding this session) — E-PR0's contract-gap-filling work will likely need to EXTEND core's `Dashboard` namespace (or `DefinitionRepository`) with a definitions-listing method; if so, that is a CORE-repo PR (its own branch, its own review loop), not admin-repo work — do not implement a workaround inside `laravel-flow-admin` that reaches into `@internal` core namespaces to fake it, per this program's `@api`/`@internal` discipline (freshly relearned the hard way in F-PR8's redaction finding this session).
- **Node catalog for the palette** (E-PR3) comes from core's `NodeRegistry::all(): array<string, NodeDefinition>` → `NodeDefinition::toArray()` (confirmed shape during F-PR7 grounding this session: `{type, name, category, icon, description, inputs, outputs, ...}`) — core likely already exposes this as a JSON endpoint for Studio-catalog purposes (verify; if not, that is again core-repo work, not an admin-repo workaround).
- **Advisor/builder UI (E-PR8, formerly "F-PR9")**: F-PR8 (`laravel-flow-ai`, merged into Macro F 2026-07-14) built `Advisor\FlowAdvisor`/`Suggestion`/`Finding` with a JSON-serializable rationale shape and `flow:suggest`/`flow:improve` CLI commands — deliberately designed "with a UI consumer in mind even though no UI exists yet" per that PR's own docblock. E-PR8 is that UI consumer arriving. `FlowBuilderService` (F-PR7) is the AI Flow Builder's (design brief §6.11) backing service — natural-language prompt → validated `GraphDefinition` draft, single-shot, typed failure on invalid output. Both live in `laravel-flow-ai`, not core — `laravel-flow-admin` will need `laravel-flow-ai` as an additional Composer dependency (currently it likely only depends on core) alongside whatever HTTP endpoints admin's own controllers expose to the React island.
- **Playwright is already wired** in this repo (`playwright.config.js`, `test-results/`, `playwright-report/` present from prior runs) — reuse the existing config/fixture conventions, do not re-architect the E2E harness from scratch.
